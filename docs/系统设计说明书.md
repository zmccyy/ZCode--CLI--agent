## 1 引言

### 1.1 编写目的

本系统设计说明书旨在全面、系统地描述 ZCode CLI Agent 的软件架构与详细设计方案，为项目组成员提供统一的技术参考和开发指导。文档整合了概要设计与详细设计内容，覆盖系统架构、核心模块、关键算法、数据结构及接口定义，确保项目开发的连贯性与一致性。

### 1.2 项目概述

ZCode CLI Agent 是一个基于终端的 AI 编程助手，对标 Claude Code v2.1.11 的核心能力。系统以 Windows 10/11 为目标平台，通过自然语言驱动软件工程任务的自动化执行。其核心能力包括：

- **自然语言交互**：用户在终端中输入自然语言指令，系统理解意图并执行相应操作
- **工具调用驱动**：通过文件读写、命令执行、代码搜索等工具链完成复杂任务
- **多模型支持**：兼容 Anthropic 第一方 API、AWS Bedrock、Google Vertex AI、Azure Foundry 及 OpenAI 兼容协议
- **MCP 扩展**：通过 Model Context Protocol 接入外部工具服务器，实现能力扩展
- **会话持久化**：支持会话恢复、上下文压缩与多层记忆系统
- **权限控制**：细粒度的工具执行权限管理，保障系统安全

## 2 系统总体设计

### 2.1 系统架构设计

ZCode 采用经典的分层架构设计，自顶向下分为五个核心层次，各层之间通过明确的接口进行交互，实现了关注点分离与松耦合。

系统整体遵循以下架构原则：

1. **单向依赖原则**：上层依赖下层，下层不依赖上层
2. **接口隔离原则**：层间通过定义清晰的接口交互，而非直接引用实现
3. **开闭原则**：工具、提供者、命令等通过注册机制扩展，无需修改核心框架
4. **特性门控原则**：通过 `feature()` 函数和 `bun:bundle` 实现编译时特性门控，消除不需要的代码路径

### 2.3 各层职责说明

#### 2.3.1 表现层 (Presentation Layer)

表现层负责用户交互与信息展示，是系统与用户之间的界面。

| 组件 | 职责 |
|---|---|
| **CLI 入口** (`entrypoints/cli.tsx`) | 解析命令行参数，执行快速路径（如 `--version`），初始化启动流程 |
| **REPL 交互** (`screens/REPL.tsx`) | 管理交互式对话循环，处理用户输入、消息展示、快捷键绑定 |
| **Ink TUI 渲染** (`components/*`, `ink/*`) | 基于 React Ink 框架渲染终端 UI，包括消息列表、权限对话框、状态栏等 |
| **命令系统** (`commands.ts`, `commands/*`) | 注册与分发斜杠命令（如 `/help`, `/model`, `/compact`），提供命令补全 |

**关键设计决策**：
- 使用 React + Ink 实现声明式终端 UI，组件化渲染消息、权限请求、状态指示器
- CLI 入口采用动态导入策略，`--version` 等快速路径零模块加载，保证启动速度
- 命令系统通过注册机制扩展，新增命令只需在 `commands/` 目录添加模块

#### 2.3.2 业务逻辑层 (Business Logic Layer)

业务逻辑层是系统的核心，负责对话循环驱动、工具调度、权限判断、上下文管理等关键业务。

| 组件 | 职责 |
|---|---|
| **查询引擎** (`QueryEngine.ts`, `query.ts`) | 驱动主对话循环：构建消息 → 调用 LLM → 解析响应 → 调度工具 → 处理结果 → 循环 |
| **工具系统** (`tools.ts`, `Tool.ts`, `tools/*`) | 管理工具注册、过滤、权限校验、执行与结果收集 |
| **权限控制** (`utils/permissions/*`) | 判断工具调用是否被允许，支持自动允许、交互确认、拒绝等模式 |
| **上下文管理** (`services/compact/*`) | 监控 token 使用量，触发自动压缩，维护对话历史摘要 |
| **任务管理** (`Task.ts`, `tasks/*`) | 管理子代理任务（本地 Bash、本地 Agent、远程 Agent、队友协作等） |
| **MCP 客户端** (`services/mcp/client.ts`) | 连接外部 MCP 服务器，发现与调用 MCP 工具，管理生命周期 |

**关键设计决策**：
- QueryEngine 采用迭代循环模式（非递归），每次迭代包含完整的"请求-响应-工具执行"周期
- 工具系统通过 `Tool` 接口统一抽象，内置工具与 MCP 工具共享同一调度路径
- 权限决策采用多级策略链：配置规则 → 分类器 → 交互确认

#### 2.3.3 数据访问层 (Data Access Layer)

数据访问层封装了与外部 LLM 服务的通信逻辑，屏蔽不同提供者的协议差异。

| 组件 | 职责 |
|---|---|
| **提供者适配** (`providers/runtime.js`) | 根据环境变量解析提供者模式，创建对应的提供者适配器 |
| **Anthropic 提供者** (`providers/anthropic.js`) | 适配 Anthropic 第一方 API、Bedrock、Vertex、Foundry |
| **OpenAI 兼容提供者** (`providers/openaiCompatible.js`) | 适配 OpenAI 兼容协议（DeepSeek 等），处理请求/响应格式转换 |
| **模型注册表** (`providers/modelRegistry.js`) | 管理可用模型列表，支持模型别名解析 |

**关键设计决策**：
- 采用适配器模式统一不同 LLM 提供者的接口差异
- 提供者选择通过环境变量驱动，支持运行时切换
- OpenAI 兼容提供者将 Anthropic 工具调用格式转换为 OpenAI Chat Completions 格式

#### 2.3.4 基础设施层 (Infrastructure Layer)

基础设施层提供底层服务支撑，包括存储、网络、配置、遥测等。

| 组件 | 职责 |
|---|---|
| **会话存储** (`utils/sessionStorage.ts`) | 持久化对话记录，支持会话恢复 |
| **API 客户端** (`services/api/client.ts`, `claude.ts`) | 封装 Anthropic SDK 调用，处理认证、流式响应、错误重试 |
| **遥测分析** (`services/analytics/*`) | 采集使用指标，上报事件，支持 GrowthBook 特性门控 |
| **文件系统操作** (`utils/file.ts`, `utils/fsOperations.ts`) | 文件读写、搜索、监控等底层文件操作 |
| **Bridge 远程管理** (`bridge/*`) | 管理远程会话、JWT 认证、工作密钥、容量唤醒 |
| **配置管理** (`utils/config.ts`, `utils/settings/*`) | 管理全局配置、项目配置、MDM 托管配置 |

### 2.4 层间交互关系

```
表现层 ──调用──→ 业务逻辑层
  │                    │
  │  用户输入           │  工具调用请求
  │  命令分发           │  LLM 请求
  │  状态展示           │  权限查询
  │                    │
  │                    ↓
  │              数据访问层
  │                    │
  │                    │  API 请求
  │                    │  模型选择
  │                    │  格式转换
  │                    │
  │                    ↓
  │              基础设施层
  │                    │
  │  状态通知           │  HTTP 请求
  │  配置读取           │  文件 I/O
  │  事件上报           │  数据持久化
  │                    │
  └────────────────────┘
```

**关键交互流程**：

1. **用户输入 → LLM 响应**：表现层接收输入 → 业务逻辑层构建消息 → 数据访问层调用 LLM → 结果逐层返回
2. **工具调用流程**：LLM 返回工具调用请求 → 业务逻辑层权限校验 → 表现层展示确认（如需）→ 业务逻辑层执行工具 → 结果回传 LLM
3. **上下文压缩**：业务逻辑层检测 token 超限 → 触发压缩算法 → 数据访问层调用 LLM 生成摘要 → 业务逻辑层替换消息历史

### 2.5 技术选型

| 类别 | 技术选择 | 选型理由 |
|---|---|---|
| 运行时 | Bun | 原生 TypeScript 支持，快速启动，内置 bundle 与 feature gate |
| 语言 | TypeScript | 类型安全，提升大型项目可维护性 |
| UI 框架 | React + Ink | 声明式终端 UI，组件化开发 |
| LLM SDK | @anthropic-ai/sdk | 官方 SDK，流式响应支持 |
| MCP SDK | @modelcontextprotocol/sdk | MCP 协议官方实现 |
| 状态管理 | React Context + 自定义 Store | 轻量级，避免引入额外依赖 |
| 命令解析 | Commander.js | 成熟的 CLI 框架，类型安全 |
| 测试 | Node.js 内置 test runner | 零依赖，原生支持 |

#### 3.1.3 命令注册与分发机制

命令系统采用集中注册 + 动态加载的模式：

1. **注册阶段**：`commands.ts` 中的 `getCommands()` 函数收集所有命令定义
2. **过滤阶段**：`filterCommandsForRemoteMode()` 根据运行模式过滤可用命令
3. **分发阶段**：用户输入 `/command` 时，匹配命令名并调用 `call()` 方法
4. **扩展机制**：新增命令只需在 `commands/` 目录创建模块并导出 `Command` 对象

## 4 关键算法与逻辑

### 4.1 对话循环驱动算法

#### 算法原理

对话循环是 ZCode 的核心驱动机制，采用迭代式循环而非递归实现。每次迭代包含"构建请求 → 调用 LLM → 解析响应 → 执行工具"四个阶段，当 LLM 返回 `end_turn` 停止原因或达到最大迭代次数时终止。

#### 实现步骤

```
算法: 对话循环驱动 (Conversation Loop)

输入: userMessage, tools, systemPrompt, maxIterations
输出: StreamEvent 流

1. 初始化消息历史 messages = [systemPrompt, ...history, userMessage]
2. iteration = 0
3. WHILE iteration < maxIterations DO
4.   规范化消息 (normalizeMessagesForAPI)
5.   构建 API 请求参数 (model, messages, tools, betas)
6.   发起流式 API 调用
7.   解析流式响应:
8.     FOR EACH event IN stream DO
9.       IF event.type == "content_block_delta" THEN
10.        IF block.type == "text" THEN 输出文本增量
11.        IF block.type == "tool_use" THEN 记录工具调用
12.        IF block.type == "thinking" THEN 记录推理增量
13.      END IF
14.    END FOR
15.  IF stop_reason == "end_turn" THEN BREAK
16.  IF stop_reason == "tool_use" THEN
17.    FOR EACH toolCall IN toolCalls DO
18.      decision = 权限检查(toolCall)
19.      IF decision.behavior == "allow" THEN
20.        result = 执行工具(toolCall.tool, toolCall.input)
21.        追加工具结果到 messages
22.      ELSE
23.        追加拒绝消息到 messages
24.      END IF
25.    END FOR
26.  END IF
27.  iteration++
28. END WHILE
29. RETURN 最终消息历史
```

#### 复杂度分析

- **时间复杂度**：O(n × m)，其中 n 为迭代次数，m 为每次迭代的工具调用数
- **空间复杂度**：O(k)，其中 k 为消息历史的总 token 数（受上下文窗口限制）
- **实际约束**：迭代次数受 `maxIterations` 限制（默认 200），token 数受模型上下文窗口限制

### 4.2 上下文自动压缩算法

#### 算法原理

当对话历史的 token 数接近模型上下文窗口上限时，系统自动触发上下文压缩。压缩通过调用 LLM 生成对话摘要，用摘要替换原始历史消息，从而释放 token 空间使对话得以继续。

#### 实现步骤

```
算法: 自动上下文压缩 (Auto Compact)

输入: messages, model, tokenCount
输出: compacted messages

1. 计算 effectiveContextWindow = contextWindowSize - reservedOutputTokens
2. 计算 autoCompactThreshold = effectiveContextWindow - bufferTokens (13000)
3. IF tokenCount < autoCompactThreshold THEN RETURN messages (无需压缩)
4. 
5. consecutiveFailures = 0
6. WHILE consecutiveFailures < MAX_CONSECUTIVE_FAILURES (3) DO
7.   分离可压缩消息与不可压缩消息
8.   构建压缩提示:
9.     - 包含系统提示
10.    - 包含对话历史（可压缩部分）
11.    - 要求 LLM 生成简洁摘要
12.  调用 LLM 生成摘要 (forked agent)
13.  IF 调用成功 THEN
14.    用摘要替换可压缩消息
15.    插入压缩边界标记 (CompactBoundaryMessage)
16.    consecutiveFailures = 0
17.    BREAK
18.  ELSE
19.    consecutiveFailures++
20.    IF 错误类型 == prompt_too_long THEN
21.      尝试更激进地截断消息后重试
22.    END IF
23.  END IF
24. END WHILE
25. 
26. 执行压缩后清理 (postCompactCleanup)
27.   - 清理文件状态缓存
28.   - 更新会话存储
29.   - 通知压缩事件
30. RETURN 压缩后的消息
```

#### 关键阈值设计

| 参数 | 值 | 说明 |
|---|---|---|
| `AUTOCOMPACT_BUFFER_TOKENS` | 13,000 | 自动压缩触发缓冲区 |
| `WARNING_THRESHOLD_BUFFER_TOKENS` | 20,000 | Token 使用量警告阈值 |
| `MAX_OUTPUT_TOKENS_FOR_SUMMARY` | 20,000 | 摘要最大输出 token 数 |
| `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES` | 3 | 连续压缩失败上限（熔断器） |

#### 复杂度分析

- **时间复杂度**：O(k)，其中 k 为待压缩消息的 token 数（需要一次 LLM 调用生成摘要）
- **空间复杂度**：O(s)，其中 s 为摘要的 token 数（通常远小于原始消息）
- **熔断机制**：连续 3 次压缩失败后停止重试，避免无限循环消耗 API 资源

### 4.3 工具权限决策算法

#### 算法原理

工具权限决策采用多级策略链模式，按照"配置规则 → 分类器 → 交互确认"的优先级依次判断，实现安全性与易用性的平衡。

#### 实现步骤

```
算法: 工具权限决策 (Permission Decision)

输入: tool, input, toolUseContext, assistantMessage, toolUseID
输出: PermissionDecision

1. 解析工具名 (getToolNameForPermissionCheck)
2. 
3. // 第一级: 拒绝规则检查
4. denyRule = getDenyRuleForTool(permissionContext, tool)
5. IF denyRule 存在 THEN
6.   RETURN { behavior: "deny", message: "工具被配置规则拒绝" }
7. END IF
8. 
9. // 第二级: 允许规则检查
10. allowRule = getAllowRuleForTool(permissionContext, tool, input)
11. IF allowRule 存在 THEN
12.   RETURN { behavior: "allow", decisionReason: { type: "config" } }
13. END IF
14.
15. // 第三级: 权限模式检查
16. IF permissionMode == "bypass" THEN
17.   RETURN { behavior: "allow", decisionReason: { type: "mode" } }
18. END IF
19. IF permissionMode == "auto-accept" THEN
20.   RETURN { behavior: "allow", decisionReason: { type: "mode" } }
21. END IF
22.
23. // 第四级: 分类器判断
24. classifierResult = checkClassifier(tool, input)
25. IF classifierResult == "auto-mode" THEN
26.   RETURN { behavior: "allow", decisionReason: { type: "classifier" } }
27. END IF
28.
29. // 第五级: 工具特定逻辑
30. IF tool.isReadOnly() AND permissionMode == "plan" THEN
31.   RETURN { behavior: "allow" }
32. END IF
33. IF tool == BashTool AND shouldUseSandbox(command) THEN
34.   RETURN { behavior: "allow", sandbox: true }
35. END IF
36.
37. // 第六级: 交互确认
38. description = tool.description(input)
39. RETURN { behavior: "ask", description }
40. // 用户在 UI 中做出选择后返回最终决策
```

#### 复杂度分析

- **时间复杂度**：O(r)，其中 r 为权限规则数量（规则匹配为线性扫描）
- **空间复杂度**：O(1)，决策过程不产生额外存储
- **性能优化**：规则匹配结果可缓存，避免重复计算

### 4.4 API 重试与退避算法

#### 算法原理

当 LLM API 调用失败时，系统采用指数退避策略进行重试。不同错误类型触发不同的退避策略：连接错误使用较长退避，速率限制使用较短退避，不可恢复错误立即失败。

#### 实现步骤

```
算法: API 重试与退避 (Retry with Backoff)

输入: apiCall, maxRetries, backoffConfig
输出: API 响应或抛出错误

1. retryCount = 0
2. WHILE retryCount < maxRetries DO
3.   TRY
4.     response = await apiCall()
5.     RETURN response
6.   CATCH error
7.     errorType = categorizeRetryableAPIError(error)
8.     
9.     IF errorType == "non_retryable" THEN
10.      THROW error  // 认证错误、参数错误等
11.    END IF
12.    
13.    IF errorType == "rate_limit" (429/529) THEN
14.      IF retryCount >= MAX_529_RETRIES (3) THEN THROW error
15.      delay = calculateBackoff(retryCount, BASE_DELAY_MS)
16.    ELSE IF errorType == "connection" THEN
17.      delay = calculateBackoff(retryCount, connInitialMs)
18.      delay = min(delay, connCapMs)
19.    ELSE  // general retryable
20.      delay = calculateBackoff(retryCount, generalInitialMs)
21.      delay = min(delay, generalCapMs)
22.    END IF
23.    
24.    IF totalDelay > giveUpMs THEN THROW error
25.    
26.    logRetryAttempt(retryCount, delay, errorType)
27.    await sleep(delay)
28.    retryCount++
29.    
30.    // 特殊处理: OAuth 401 时刷新令牌
31.    IF errorType == "auth_expired" THEN
32.      await refreshOAuthToken()
33.    END IF
34.    
35.    // 特殊处理: 连接错误时清除凭证缓存
36.    IF errorType == "connection" THEN
37.      clearCredentialsCache()
38.    END IF
39.  END TRY
40. END WHILE
41. THROW MaxRetriesExceededError

函数 calculateBackoff(attempt, baseDelay):
  RETURN baseDelay * 2^attempt + randomJitter()
```

#### 退避参数设计

| 参数 | 连接错误 | 通用错误 | 速率限制 |
|---|---|---|---|
| 初始延迟 | 2,000 ms | 500 ms | 500 ms |
| 最大延迟 | 120,000 ms | 30,000 ms | 30,000 ms |
| 放弃阈值 | 600,000 ms | 600,000 ms | 600,000 ms |
| 最大重试 | 10 | 10 | 3 (529) |

### 4.5 提供者路由与模型解析算法

#### 算法原理

系统根据环境变量和用户配置，将模型请求路由到正确的 LLM 提供者。模型解析支持别名（如 "sonnet", "opus", "haiku"）到具体模型 ID 的映射，并根据提供者模式转换模型标识符。

#### 实现步骤

```
算法: 提供者路由与模型解析 (Provider Routing & Model Resolution)

输入: userSpecifiedModel, env
输出: { provider, model, apiConfig }

1. // 步骤 1: 解析提供者模式
2. providerMode = resolveProviderMode(env)
3.   IF env.ZCODE_PROVIDER == "openai-compatible" THEN
4.     RETURN "openai-compatible"
5.   END IF
6.   IF env.ANTHROPIC_API_KEY 存在 THEN RETURN "firstParty"
7.   IF AWS 凭证配置 THEN RETURN "bedrock"
8.   IF GCP 凭证配置 THEN RETURN "vertex"
9.   IF Azure 配置 THEN RETURN "foundry"
10.  RETURN "firstParty"  // 默认

11. // 步骤 2: 解析模型标识符
12. modelSetting = getUserSpecifiedModelSetting()
13.   // 优先级: 会话内覆盖 > --model 参数 > 环境变量 > 设置
14.  IF modelSetting 是别名 (如 "sonnet") THEN
15.    modelId = resolveAlias(modelSetting, providerMode)
16.    // 例如: "sonnet" → "claude-sonnet-4-20250514" (firstParty)
17.    //       "sonnet" → "anthropic.claude-sonnet-4-20250514-v1:0" (bedrock)
18.  ELSE
19.    modelId = modelSetting
20.  END IF

21. // 步骤 3: 构建提供者适配器
22. IF providerMode == "openai-compatible" THEN
23.   adapter = createOpenAICompatibleProvider({
24.     baseUrl: env.ZCODE_OPENAI_BASE_URL,
25.     apiKey: env.ZCODE_OPENAI_API_KEY,
26.     model: env.ZCODE_OPENAI_MODEL || modelId,
27.     headers: env.ZCODE_OPENAI_HEADERS,
28.     timeout: env.ZCODE_OPENAI_TIMEOUT
29.   })
30. ELSE
31.   adapter = createAnthropicProvider({ provider: providerMode })
32. END IF

33. RETURN { provider: adapter, model: modelId, apiConfig }
```

#### 模型别名解析映射

| 别名 | First Party | Bedrock | Vertex |
|---|---|---|---|
| sonnet | claude-sonnet-4-20250514 | anthropic.claude-sonnet-4-20250514-v1:0 | claude-sonnet-4@20250514 |
| opus | claude-opus-4-20250514 | anthropic.claude-opus-4-20250514-v1:0 | claude-opus-4@20250514 |
| haiku | claude-haiku-4-5-20250514 | anthropic.claude-haiku-4-5-20250514-v1:0 | claude-haiku-4-5@20250514 |

  call(input: Input, context: ToolUseContext): Promise<string>

  renderToolUse?(input: Input): ReactNode      // TUI 渲染（可选）

}
